# Local-First Teacher Grading Assistant on iOS and iPadOS

## Executive summary

A teacher-in-the-loop grading assistant with a fully local core workflow is **technically feasible today** on Apple platforms, but only inside a narrow and explicit availability envelope. The strongest current Apple-native path is: capture or import student work, extract or review text locally with Vision/PDFKit/VisionKit, package rubric criteria and instructions into small structured prompts, and ask Foundation Models for a **draft** criterion-by-criterion outcome with explanations and evidence references that the teacher must review before finalizing. Apple’s public APIs support on-device OCR, document scanning, structured text generation, streaming, and tool calling. They do **not** publicly document multimodal Foundation Models input for images, nor do they position the on-device model as suitable for high-context, high-reasoning, or fully autonomous decision-making. citeturn33view1turn38search2turn33view3turn25search1

The main gating constraint is Foundation Models availability. The public framework is documented for **iOS 26 / iPadOS 26 and Xcode 26**, and it depends on Apple Intelligence being supported on the device, turned on in Settings, available in the user’s language/region, and actually ready on-device. Apple’s docs explicitly note that availability can be `.modelNotReady`, and they also note the model may take time to download. That means the app can be truly offline **only after** the required system assets are already present and the model reports `.available`; a strict “fresh-device, first-launch, never-online” guarantee is **not** documented. citeturn12search0turn12search8turn12search9turn12search3turn30view0

For OCR and document ingestion, Apple’s stack is stronger and more mature than the grading layer. Vision text recognition exposes recognized text, confidence, and bounding boxes; Vision’s newer `RecognizeDocumentsRequest` adds document structure such as paragraphs, tables, lists, QR codes, and important data extraction; VisionKit supplies document-camera capture and live scanning; and PDFKit can extract embedded PDF text directly and can perform on-demand local Live Text OCR for scanned PDFs. This is a very good fit for “scan/import/paste → teacher review extracted text → local draft grading,” especially for typed or OCR-readable student text. citeturn5search0turn5search2turn5search3turn33view3turn20search0turn20search3turn33view5turn25search1

The most important architectural implication is that Foundation Models should be treated as a **small, text-only, structured drafting engine**, not as the source of truth. Apple describes the on-device model as device-scale and optimized for tasks like summarization, extraction, classification, and tagging, while explicitly saying it is **not designed for world knowledge or advanced reasoning**. Combined with the documented context-window limit of **4,096 tokens**, this means whole-paper end-to-end grading in one request is the wrong architecture. The right architecture is per-criterion or otherwise heavily chunked prompting, deterministic scoring logic in app code, and strict teacher review of every proposed result. citeturn33view1turn33view2turn39search1turn38search4turn38search6

## Vision and VisionKit OCR pipeline

The recommended ingestion pipeline is Apple-native and strongly local. Use `VNDocumentCameraViewController` for multipage paper capture, `PhotosUI` and `UIDocumentPickerViewController` for image/PDF/text imports, and PDFKit first for PDFs that already contain embedded text. Only when a page has no usable embedded text should the app render the page to an image and run Vision OCR. That preserves fidelity, reduces OCR errors, and avoids wasting time on pages that are already text-based. Apple documents document-picker import, UTType-based file typing, PDF text extraction, and the document-camera page image API for this stack. citeturn23search1turn23search0turn25search1turn25search0turn20search0

For OCR itself, use Vision as the canonical programmatic engine. Apple documents `VNRecognizeTextRequest` / `RecognizeTextRequest` for image OCR, configurable recognition level, language correction, minimum text height, custom words, and supported-language queries. At the result layer, `RecognizedTextObservation` and `RecognizedText` expose recognized content, confidence, and bounding boxes, including per-range bounding boxes. This is exactly what a grading assistant needs to support teacher correction and evidence linking. citeturn18search2turn18search8turn18search1turn18search0turn18search3turn5search3turn5search0turn5search2

On iOS/iPadOS 26, add `RecognizeDocumentsRequest` as a second-stage structural pass when the page looks like a worksheet, quiz table, form, or rubric. Apple documents document observations grouped by words, lines, and paragraphs, and WWDC25 states that the API can detect tables and lists, recognize QR/barcode content, and identify items such as email addresses, phone numbers, or URLs across 26 languages. For typed school documents and structured answer sheets, that can reduce a lot of custom layout parsing. citeturn19search2turn33view3

VisionKit should be used selectively. `VNDocumentCameraViewController` is the right Apple-native UI for “scan these pages into the app,” because it yields a clean page-by-page document capture. `DataScannerViewController` is useful when a teacher wants quick live capture from the camera, but Apple explicitly requires `isSupported` / `isAvailable` checks and camera permission, and its natural fit is live detection rather than a durable, multipage grading record. `ImageAnalyzer` / `ImageAnalysisInteraction` can help with text-selection review in still images, but they are not a substitute for your own canonical OCR data model. citeturn20search3turn20search0turn4search6turn4search5turn21search1turn22search2turn22search7

The core OCR record should therefore be a normalized, page-based local model that stores: original artifact reference, page index, plain text transcript, segment ranges, bounding boxes, and confidence summaries. Because Vision gives both text and location, you can let the teacher tap a quote in the extracted transcript and highlight the source region on the original page; that is the right substrate for later “evidence quote” features in grading. This is an architectural recommendation grounded in the documented OCR result types and PDF text-selection APIs. citeturn5search0turn5search2turn5search3turn25search3turn25search5

For offline/local-only assessment, the OCR stack is materially better than the grading stack:

| OCR-related function | Local-only status | Caveats |

| Vision OCR (`VNRecognizeTextRequest`) | Apple’s Vision framework runs entirely on-device. citeturn33view3 | Quality still depends on page quality, handwriting legibility, and language support. Use supported-language queries and teacher review. citeturn18search3turn5search2 |

## Foundation Models integration notes

For grading, Foundation Models should be integrated as a **typed draft generator**. Apple’s documented structured-output features map well to a schema such as:

- `criterionID`
- `proposedBand` or `proposedPointsBucket`
- `evidenceSpanIDs`
- `explanation`
- `uncertaintyLevel`
- `uncertaintyReason`
- `draftStudentFeedback`

Apple’s `@Generable` and `Guide` macros, along with `count`, `range`, `constant`, `anyOf`, and `pattern`, are useful here because they let you strongly constrain labels, required counts, and field formats, while still keeping the result in Swift-native structures. Apple further documents that guided generation enforces the schema shape and will not invent unexpected fields. citeturn31view4turn35search0turn35search1turn35search2turn35search3turn35search4turn35search5turn34view2

The best prompting pattern for this use case is **small, criterion-scoped, evidence-aware prompting**, not “grade the whole paper.” Apple’s own positioning matters here: the on-device model is optimized for summarization, extraction, classification, tagging, and similar device-scale tasks, and Apple explicitly says it is not designed for world knowledge or advanced reasoning. Apple also documents a 4,096-token context window, with inputs and outputs both counting against that budget, and explains that one token is roughly three to four English characters. In practice, that is not enough room for a long essay, a full rubric, one or more exemplars, an answer key, and a long explanation all at once. citeturn33view1turn33view2turn39search1turn38search4turn38search6

For a grading assistant, the right implementation is therefore a two-stage or three-stage local workflow. First, the app code segments and normalizes the student text. Second, it sends **one criterion at a time** or one small criterion bundle at a time to Foundation Models. Third, deterministic code outside the model totals the scores, enforces allowed point values, validates evidence references, and blocks finalization until the teacher approves. This is an inference from Apple’s documented model positioning and context limits, but it is the architecture most aligned with the APIs Apple has actually shipped. citeturn33view1turn39search1turn38search6

Use examples and exemplars, but do so economically. Apple’s code-along shows few-shot prompting, `Prompt` builders, and `includeSchemaInPrompt: false` to save context window space when you already have a good example. It also documents `prewarm()` for better responsiveness and shows greedy sampling in structured/tool-assisted generation. Those patterns transfer well to rubric grading: one or two carefully chosen exemplar outputs, greedy sampling for stability, prewarming when the grading screen opens, and explicit field validation after every call. citeturn31view0turn32view0turn32view2turn40search1

Tool calling is documented, but it should be treated cautiously in this product. Apple documents that the model decides whether and how often to call a tool; tools can be invoked multiple times and in parallel. That makes tool calling useful for **optional** local helpers, such as retrieving a locally stored evidence span by ID or looking up rubric metadata, but inappropriate for any mandatory business logic. Do not make “use the tool to compute the official score” a required step. The score calculation, aggregation, and validation belong in your own code, not behind tool invocation. citeturn12search5turn34view0turn34view3

Custom adapters exist, but they are a bad fit for the core grading flow. Apple’s adapter docs say that adapters are version-specific to the system model, require a deployment entitlement, add about 160 MB each, and should generally be delivered from a server via Background Assets rather than bundled. That directly conflicts with a no-cloud core path. If the product later needs domain specialization, try to reach acceptable quality first with prompt engineering, guides, and local code. Treat adapters as a later experiment, not as part of the initial feasibility case. citeturn9view11turn29search7

What should **not** be built on Foundation Models in the first version is equally important. Do not use it to compute final totals, to decide whether a criterion is satisfied without evidence validation, to understand source images directly, to grade diagrams or artworks, or to dependably reason over very long papers. Apple’s public docs do not document image inputs for third-party Foundation Models usage, and Apple explicitly frames the model as text-based and not intended for advanced reasoning. citeturn26search0turn26search2turn33view1

The core application model should be **artifact-first**. Persist the original asset locally, then derived OCR pages, then normalized text spans, then rubric definitions, then generation requests and responses, then teacher-approved final results. That structure is not just good software design; it is what makes teacher review, re-OCR, prompt replays, and auditability possible while staying offline and privacy-preserving. The rationale is grounded in the fact that Apple’s OCR stack exposes page-level text and locations, and Foundation Models sessions are small and stateful rather than durable sources of record. citeturn5search0turn25search5turn16search12

The main implementation risks are not “can Apple’s APIs do something at all,” but “can they do it consistently enough for this product promise?” The biggest risks are: small FM context windows, model-availability gating outside your app’s control, undocumented variance across model versions, poor or messy handwriting, and the temptation to over-trust generated explanations. Apple’s own docs note current model-version variation across OS 26 releases, and adapter compatibility is explicitly tied to system model versions. That means prompt regressions across OS updates are not hypothetical; they are part of the platform contract. citeturn12search22turn9view11

There is also a non-technical platform risk. Apple’s Foundation Models acceptable-use requirements prohibit a number of sensitive or regulated uses and separately prohibit generating courseware and certain academic products. A teacher-review-only grading assistant is **not expressly listed** as prohibited in the text I reviewed, but any feature drift toward autonomous educational decision-making or courseware generation deserves an App Review / policy read before launch. citeturn36search0turn36search1

For later features, the likely boundary is clear. Apple’s currently documented document APIs cover text extraction, document structure, tables, and machine-readable codes. I did **not** find Apple documentation for Foundation Models image input, math-notation OCR, or a public Apple-native API that would reliably grade diagrams, posters, physical models, or general visual artifacts. Those should be treated as **Core ML or other local-model** territory, potentially combined with Vision pre/post-processing. citeturn26search0turn26search2turn33view3turn8search0turn8search5

## Validation checklist and product claims
