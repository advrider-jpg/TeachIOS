## MVP scope and deferred work

The MVP should optimize for the **single-submission grading loop**, because that is the most defensible, most teacher-controlled, and best-supported workflow across established systems. Canvas SpeedGrader centers opening one submission, viewing the rubric, clicking criterion ratings, entering criterion comments, and saving; Google Classroom lets teachers create, reuse, and export rubrics; Gradescope likewise centers the submission area, a rubric pane, and navigation to the next ungraded item. In other words, the marketŌĆÖs mature grading UX is built from **one reviewed submission at a time**, with reusable rubric structure surrounding it. ŅłĆciteŅłéturn24view1Ņłéturn24view0Ņłéturn24view2Ņłü

The MVP should therefore include six capabilities and stop there. First, **local capture/import** from camera, photos, files, and PDFs. Second, **local OCR** plus a teacher-editable transcript. Third, **structured rubric templates** with criteria, levels, points, free-form notes, and saved reuse. Fourth, **draft criterion scoring with evidence quotes** and an explicit **uncertainty state** when OCR confidence or evidence strength is weak. Fifth, **teacher final approval** before any grade becomes exportable. Sixth, **local export** to PDF, CSV, and a restorable archive. That combination directly matches the highest-confidence market need while staying faithful to the core promise of local-first, teacher-controlled grading. This is a strategic inference built from the observed assessment workflows and privacy/capture benchmarks. ## Export, backup, and privacy hardening

The MVP export strategy should lead with **PDF**, **CSV**, and a **ZIP-based archive**. That choice best matches what teachers and adjacent tools already use. PDF is the default human-readable output across scanner apps and teacher report tools, and iDoceo explicitly exports configurable reports in PDF. CSV is the most common grade interchange format across Canvas and Google Classroom, while Gradescope supports CSV and Excel export. A ZIP-wrapped local archive is the right answer for backup and restore, not for daily sharing, because it lets you preserve the full grading record without flattening it into a spreadsheet or presentation format. ŅłĆciteŅłéturn11view0Ņłéturn35view1Ņłéturn35view3Ņłéturn8search1Ņłéturn8search5Ņłéturn8search10Ņłéturn8search19Ņłü

For the MVP, the practical ranking is: **PDF first, CSV second, ZIP archive third, XLSX later, DOCX/RTF not now**. PDF is best for student-facing or parent-facing feedback packets because it preserves layout, comments, rubric summaries, and evidence excerpts. CSV is best for gradebook ingestion because LMS tools already expect it. The archive is best for recovery and portability because it can store raw submissions, OCR text, rubric state, and exports together. XLSX can wait because it is valuable mainly for spreadsheet power users and some reporting workflows, and the best-reviewed pure Swift XLSX package in this research set, CoreXLSX, is explicitly a parser rather than a writer. DOCX and RTF should be deferred because I did not find them acting as core interchange formats in the teacher grading workflows reviewed here. ŅłĆciteŅłéturn8search1Ņłéturn8search10Ņłéturn8search19Ņłéturn11view0Ņłéturn37view2Ņłü

Spreadsheet export needs explicit security work. OWASP and CWE both treat formula injection as a real weakness in CSV and spreadsheet-compatible exports. The dangerous prefixes include `=`, `+`, `-`, `@`, tab, carriage return, line feed, and even some full-width variants; attackers can also exploit separators and quotes to move a dangerous prefix to the beginning of a cell. OWASPŌĆÖs current guidance is that normal quote-escaping alone is not enough for Microsoft Excel after save/reopen, and that a tab prefix inside a quoted field is the most Excel-resistant mitigation for human-view spreadsheet files, though it changes the underlying data. ŅłĆciteŅłéturn30view0Ņłéturn30view1Ņłéturn30view2Ņłü

That means the MVP should ship **two distinct spreadsheet philosophies**, not one ambiguous CSV. For **gradebook import CSV**, keep the file narrow and mostly numeric: internal IDs, assignment identifiers, criterion totals if needed, final approved score, and perhaps short status fields. Free-form narrative comments and evidence excerpts should stay out of that import file by default. For **human-view CSV**, sanitize every untrusted field per OWASP guidance, quote every field, escape embedded quotes, and neutralize formula-triggering prefixes. If you do not want a second CSV mode, the safer MVP choice is even simpler: keep detailed comments and evidence in **PDF and archive JSON only**, and reserve CSV for minimal grade transfer. That recommendation is an inference, but it follows directly from the OWASP tradeoffs and the LMS import/use patterns in Canvas and Google Classroom. ŅłĆciteŅłéturn30view0Ņłéturn30view1Ņłéturn8search1Ņłéturn8search5Ņłéturn8search10Ņłü

The local backup/archive format should be a **ZIP bundle with a clear manifest and a consistent database snapshot**, not a loose folder copy. SQLiteŌĆÖs online backup API exists for live backups, and SQLite documents both the online backup API and `VACUUM INTO` as supported ways to create backup copies of a live database. SQLiteŌĆÖs WAL documentation also explains that a checkpoint moves WAL transactions back into the main database, which is why na├»vely copying a `.db` file is a poor backup strategy once WAL is involved. ŅłĆciteŅłéturn31search1Ņłéturn31search8Ņłéturn31search15Ņłéturn31search2Ņłü

A good archive bundle should include: **`manifest.json`** with app version, schema version, export timestamp, and checksum list; **a single SQLite snapshot** created via proper backup mechanics; **original imported files** such as PDFs or images; **OCR text and teacher-corrected text**; **rubric definitions and rubric version identifiers**; **criterion-level draft and final decisions**; **evidence quote anchors** based on character ranges or paragraph indices; **uncertainty flags and reasons**; and **any generated PDFs or CSVs**. ## Architecture and dependency stack

The persistence decision should be **SQLite with GRDB.swift, not SwiftData**, for this product. SwiftData is attractive when you want quick Swift-native persistence and Apple-managed modeling, and Apple documents schema objects and migration plans for it. But GRDB gives you direct SQLite control, schema migrations, observation, and raw SQL access, and positions itself as a SQLite toolkit for application development. For a local-first grading app that needs deterministic backups, portable archives, explicit migrations, robust export logic, and likely full-text search or quote anchoring later, the extra control is a feature, not a burden. This is a recommendation, but it is based directly on the relative capabilities exposed by AppleŌĆÖs SwiftData docs and GRDBŌĆÖs documented scope. ŅłĆciteŅłéturn22search2Ņłéturn22search0Ņłéturn22search17Ņłéturn22search4Ņłéturn22search11Ņłéturn20view0Ņłéturn20view1Ņłü

For OCR and capture, the right starting point is **AppleŌĆÖs native stack**, not an open-source OCR dependency. VisionKit is explicitly for analyzing pixel information and isolating important data such as text, while Vision exposes OCR through `VNRecognizeTextRequest`; Apple also documents live text/data scanning in VisionKit. If you stay here, your core OCR can remain local and aligned with the productŌĆÖs privacy promise. ŅłĆciteŅłéturn28search0Ņłéturn28search1Ņłéturn28search16Ņłéturn28search13Ņłü

For PDF generation, the best split is **custom generation with `UIGraphicsPDFRenderer`** and optional **PDFKit for preview or annotation display**, not TPPDF as the default export engine. Google ClassroomŌĆÖs rubric model is explicitly structured around criterion titles, criterion descriptions, level titles, level descriptions, and points. Canvas and Gradescope likewise treat rubrics as structured rating/comment objects. AppleŌĆÖs `swift-markdown` package is high quality for parsing, building, editing, and analyzing markdown documents, but markdown is still text-first.
